# Print AR Module

`src/modules/print-ar/` is the canonical reusable AR preview package for framed prints and artworks inside this project.

## What This Module Is

- reusable customer-facing AR preview module for framed print variants
- Android-first for practical wall preview
- iPhone/iPad Quick Look preserved as a secondary fallback
- local-first and pre-generated by design

This module freezes the agreed print model:

- paper size is the source of truth
- print borders are already included in paper size
- no crop
- frame and mat logic stay unchanged

## What A Host Site Must Provide

At minimum, a host integration should provide:

- print or product id
- title or label
- artwork image URL
- paper size selection
- frame preset id
- mat data
- share URL or product URL when customer CTA needs it

The payload builder currently expects a print-like input with:

- `id`
- `title`
- `image`
- `sizes`
- optional `series`
- optional `edition`
- optional `paper`
- optional `priceFrom`
- optional `buyUrl`

## Exact Variant Identity

Variant identity is strict.

- use `buildPrintArAssetKey()` or `buildVariantAssetId()`
- frame preset is part of exact variant identity
- current asset key shape is:
  - `<printId>-<sizeKey>-<framePreset>-frame-warm-white-mat`
- one exact variant maps to one exact hosted GLB
- one exact variant may also map to one hosted USDZ

Hosted asset binding rules:

- customer Android CTA must only launch when the exact hosted GLB variant is attached and reachable
- iPhone/iPad fallback must only reflect the exact hosted USDZ binding for that same asset key
- manifest status stays platform-specific and honest

## What The Host Site Consumes

Normal customer flow should consume only:

- `PrintArOverlay` or `CustomerPrintArOverlay`
- payloads built by `buildPrintArPayload()`
- ready-only preview state
- Android primary launch path
- iPhone/iPad fallback path

Typical host flow:

1. Build payload for the selected print, size, and frame preset.
2. Open the overlay through the module event bridge or overlay API.
3. Let the customer overlay validate hosted exact-variant readiness.
4. Offer Android launch only when the exact variant is ready.
5. Keep iPhone/iPad as fallback only.

## What Is Internal Only

Internal-only surfaces are not part of customer mode:

- generation workflow panel
- local GLB generation
- local USDZ generation
- manifest prep helpers
- attach/export operations
- internal diagnostics and readiness review

## Current Operating Model

- local-first pre-generation is the current operating model
- no backend is required right now
- Android remains the primary practical wall-preview path
- iPhone/iPad remains secondary fallback
- the next separate system is an Internal AR Prep Tool
- native iOS wall-preview remains future R&D, not current package scope
